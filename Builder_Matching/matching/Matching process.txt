GDELT–Reference Flood Matching Pipeline
========================================

Overview
--------
The matching pipeline links GDELT news-derived flood events to validated
reference flood records (DFO, EM-DAT, GDACS, etc.), then propagates impact
fields (deaths, displaced, damage) from the reference onto the matched GDELT
events.  This enriches the GDELT database with ground-truth severity data
while preserving its news-coverage characteristics.

The pipeline runs in 9 sequential stages defined in run_matching.py.


Input Files
-----------
  Builder_Reference/outputs/reference_floods_enriched.jsonl
      ~6,750 consolidated flood records from DFO, EM-DAT, GDACS, ReliefWeb,
      Copernicus EMS, Desinventar, HANZE, and IFRC — cross-source deduplicated
      and enriched with hydro-climate indicators.

  Builder_GDELT/outputs/gdelt_floods_enriched.jsonl
      ~775 GDELT flood events (Will be more once we get the API credits) extracted from news articles via LLM (gpt-5-mini),
      geocoded from GDELT URL metadata, and enriched with the same hydro-climate
      indicators as the reference dataset.

Output Files
------------
  Builder_GDELT/outputs/matching/gdelt_floods_matched.jsonl
      All GDELT flood events with matched=True/False.  Matched events also carry
      ref_* prefixed fields propagated from the best-matching reference record.

  Builder_GDELT/outputs/matching/evaluation_report.json
      Precision, recall, F1, and breakdowns by year and reference source.

Usage
-----
  # Full run
  python -m Builder_Matching.matching.run_matching

  # Restricted to a specific date window (recommended for preliminary testing)
  python -m Builder_Matching.matching.run_matching --date-from 2018-01-01 --date-to 2018-02-28

  # Tune parameters
  python -m Builder_Matching.matching.run_matching --max-km 200 --max-days 7 --threshold 0.45

  Parameters:
    --max-km     Spatial filter radius in km (default: 300)
    --max-days   Temporal tolerance in days on each side of ref date range (default: 14)
    --threshold  Minimum match score to accept a pair (default: 0.40)
    --date-from  Filter both datasets to events on/after YYYY-MM-DD
    --date-to    Filter both datasets to events on/before YYYY-MM-DD


Pipeline Stages
---------------

Stage 1 — Load
  Read both JSONL files.  GDELT events are filtered to disruption_type="flood".
  Optional --date-from/--date-to filters are applied to both datasets.

Stage 2 — Candidate Generation  (candidate_generation.py)
  Avoids an O(N²) all-pairs comparison by applying two cheap pre-filters:

  Temporal filter:
    The GDELT event date must fall within the reference event's date range,
    extended by ±max_days on each side.  This is an overlap check (not just
    proximity), so long-duration floods (e.g. 90-day monsoon events) are
    correctly paired with GDELT articles published at any point during the
    flood.  The ±14-day tolerance accounts for reporting lag.

  Spatial filter:
    Haversine distance between event centroids must be ≤ max_km (default 300 km).
    300 km accommodates geocoding imprecision (Desinventar admin-1 centroids can
    be 100+ km off) while excluding events in neighbouring countries.

  Together these filters typically reduce candidate pairs by >99%.

Stage 3 — Feature Vectors  (feature_vectors.py)
  Builds a 7-dimensional hydro-climate feature matrix for each event:
    1. spi_30d                  Standardized Precipitation Index (30-day)
    2. glofas_discharge_pct_rank  River discharge percentile rank
    3. chirps_7d_anom_pct       7-day rainfall anomaly (% above climatology)
    4. gpm_peak_3h_mm           Peak 3-hour rainfall intensity
    5. era5_soil_moisture_day0  Antecedent soil moisture
    6. jrc_recurrence_pct       Surface water recurrence (floodplain signal)
    7. pop_density_km2          Population density within 25 km

  Missing values are imputed with the dataset median.  Vectors are normalised
  before computing cosine similarity.

Stage 4 — Scoring  (scoring.py)
  Each candidate pair receives a weighted similarity score in [0, 1]:

    Component           Weight   Method
    ------------------  ------   ----------------------------------------
    Geo distance         0.30    exp(-d / 100 km); 1.0 at 0 km, 0.37 at 100 km
    Temporal proximity   0.20    1.0 inside ref date range; 1/(1+days_outside) outside
    Hydro fingerprint    0.35    Cosine similarity of 7-D feature vectors
    Location text        0.15    Jaccard token overlap on country + location_name

  The hydro fingerprint carries the highest weight because it is an
  objective, source-independent physical signal — two events in the same
  region and week could be different floods, and only the hydro fingerprint
  can distinguish them.

Stage 5 — Graph Construction  (graph_builder.py)
  Builds a bipartite NetworkX graph:
    Left  nodes: GDELT events
    Right nodes: Reference events
    Edges: candidate pairs with score ≥ 0.25 (permissive threshold to
           preserve weak edges for global optimisation)

Stage 6 — Maximum-Weight Bipartite Matching  (matching.py)
  Runs NetworkX max_weight_matching() (Galil-Micali-Gabow algorithm) to find
  a globally optimal 1-to-1 assignment.  Each GDELT event is assigned to at
  most one reference event and vice versa.  Pairs below the acceptance
  threshold (default 0.40) are discarded even if they are the best available.

Stage 7 — Richness Propagation  (matching.py)
  For each matched GDELT event, 17 reference fields are copied across with a
  ref_ prefix:
    Impact:   dead, injured, displaced, affected, indirectly_affected,
              houses_destroyed, houses_damaged, roads_km, damage_usd_thousands
    Metadata: severity, main_cause, event_name, area_km2, glide_number,
              date_end, country_iso, all_country_iso
  Match provenance fields are also added:
    matched, match_score, match_ref_source, match_ref_id, match_ref_date

Stage 8 — Evaluation  (evaluation.py)
  Computes precision, recall, and F1:
    Precision = matched GDELT / total GDELT  (how accurate is GDELT?)
    Recall    = matched reference / total reference  (how much does GDELT cover?)
  Breakdowns by year and reference source are included in evaluation_report.json.

Stage 9 — Write outputs
  Writes gdelt_floods_matched.jsonl and evaluation_report.json.


Interpreting Results
--------------------
Precision measures how often a GDELT flood article corresponds to a
catalogued significant flood event.  Low precision (~1-5%) is expected
because GDELT captures all reported floods including minor local events
that do not appear in curated databases.

Recall measures how much of the reference database GDELT covers.  With
only Jan-Feb 2018 data available, recall against the full multi-year
reference database is low by construction.  Restricting to the same date
window (--date-from / --date-to) gives a fairer evaluation:

  Full run (Jan-Feb 2018 GDELT vs all years reference): recall = 0.003
  Date-filtered (Jan-Feb 2018 vs Jan-Feb 2018 reference): recall = 0.156

The 15.6% within-window recall indicates the matcher is functioning
correctly.  As more GDELT months are scraped, recall will improve.
